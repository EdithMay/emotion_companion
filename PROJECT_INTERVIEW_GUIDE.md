# PROJECT_INTERVIEW_GUIDE.md

## 1. 这到底是一个什么项目

这个项目可以理解成一个“有记忆能力的情绪陪伴 Agent”。它不是传统意义上的客服问答，也不是只把用户输入转发给大模型的聊天壳，而是围绕一个具体场景做的 AI 应用：用户在生活、学习、工作中有情绪波动时，可以在这里倾诉；系统会以不同陪伴风格回应，并把长期对话沉淀为心情记录、记忆摘要和可检索的长期记忆。

如果面试时用一句话介绍，可以这样说：

> 我做的是一个情绪陪伴类 AI 应用，核心是让用户可以持续和一个有记忆感的 Agent 对话。后端用 FastAPI 承接接口，用数据库保存会话和消息，用 ChromaDB 做长期语义记忆，用大模型完成陪伴回复、记忆压缩和心情总结；前端用 Vue3 做聊天界面、会话侧栏和心情日历，让整个项目从“能聊”变成“能长期陪伴和复盘情绪”。

这个项目真正想解决的问题有三个：

1. 用户不是只需要一次性答案，而是需要连续、自然、有上下文的陪伴。
2. 大模型默认没有长期记忆，所以需要把重要历史沉淀下来，并在合适的时候召回。
3. 情绪类产品不能只停留在聊天，还应该把对话转化成可观察的心情变化，比如每日总结、分数、关键词、趋势和日历。

所以这个项目的价值不只是“调用了 LLM”，而是把大模型能力放进了一个完整产品闭环里：对话、记忆、总结、可视化、主动关怀、工具增强。

## 2. 面试时可以怎么开场介绍

### 1 分钟版本

我这个项目是一个情绪陪伴类 AI Agent，名字叫“树洞”。用户可以创建多个会话，选择不同陪伴风格，然后像聊天一样倾诉。系统会保存对话历史，并且通过短期上下文、中期摘要和长期向量记忆，让 Agent 在后续对话中能够记住用户之前提到过的重要事情。

技术上，后端我用 FastAPI 做接口，用 SQLAlchemy 和 SQLite 存会话、消息、心情记录和人设配置，用 ChromaDB 加 DashScope Embedding 做长期记忆检索，用大模型生成回复、压缩记忆和分析每日心情。前端用 Vue3 和 TypeScript 做聊天页、会话侧栏、心情日历、天气和热点话题组件。这个项目比较能体现我对 LLM 应用开发的理解，因为它不是单次问答，而是有上下文、有记忆、有流式输出、有数据沉淀的完整 AI 应用。

### 3 分钟版本

这个项目的业务背景是情绪陪伴。很多聊天机器人只会回答当前问题，但情绪陪伴更重要的是连续性和关系感：用户今天说了工作压力，过几天又提到同一个同事，系统应该能在合适的时候想起之前的背景；用户连续几天心情很低落时，系统也应该在 Prompt 层做更温和的主动关怀。

所以我把项目拆成几个核心模块。第一是对话模块，负责会话创建、消息保存和流式回复。用户发消息后，后端会先把用户消息存入数据库，再读取最近几条消息和之前压缩过的摘要，构建 Prompt 调用大模型。第二是记忆模块，最近几条原始消息作为短期记忆，较早的消息会被 LLM 压缩成中期摘要，重要的用户消息、心情总结和摘要会写入 ChromaDB 作为长期语义记忆。第三是 Agent 工具调用模块，模型不是每次都强制检索记忆，而是先判断是否需要调用 `search_memory`，如果需要就检索 ChromaDB，再把结果作为背景生成最终回复。第四是心情总结模块，它会读取当天用户说过的话，让大模型输出情绪分数、关键词和总结，并保存到数据库，前端再展示成日历、近 7 天趋势和关键词云。

后端我用 FastAPI 做接口层，业务逻辑放在 service 和 Agent 中，数据库用 SQLite 保存结构化数据，用 ChromaDB 保存向量记忆。前端用 Vue3 和 TypeScript，普通接口用 axios，聊天流式接口用 fetch 读取 ReadableStream，这样 AI 回复可以边生成边展示。项目的难点主要是多轮上下文管理、长期记忆召回、流式响应、LLM 输出 JSON 的稳定性，以及关系型数据库和向量库之间的一致性。后续如果继续优化，我会补用户体系、按用户隔离 RAG、加入后台任务队列、统一异常处理，并扩展文档知识库上传能力。

## 3. 面试官听不见代码时，项目架构应该怎么讲

你可以把整个项目讲成一条数据流：

```text
用户在前端输入消息
    ↓
前端把会话 ID 和消息内容发给后端
    ↓
FastAPI 接口接收请求并校验参数
    ↓
后端先保存用户消息
    ↓
Agent 读取最近对话、会话摘要、人设配置和必要的长期记忆
    ↓
Agent 构建 Prompt 调用大模型
    ↓
如果模型判断需要查记忆，就调用 RAG 检索历史内容
    ↓
模型生成回复，后端以 SSE 流式返回前端
    ↓
前端逐字追加显示，同时后端保存 AI 回复
    ↓
有价值的用户消息继续写入向量库，形成长期记忆
```

从架构上看，它不是“前端直接问大模型”，而是：

```text
前端交互层
    ↓
API 请求层
    ↓
FastAPI 路由层
    ↓
业务服务层
    ↓
Agent 编排层
    ↓
数据库 / 向量库 / 大模型 / 第三方工具
```

每一层有明确职责：

前端负责用户体验，比如会话列表、消息气泡、输入框、流式渲染、心情日历、天气卡片和热点话题。

接口层负责接收请求，比如创建会话、发送消息、生成心情总结、查询天气和新闻。

业务层负责真正的逻辑，比如保存消息、压缩记忆、生成心情总结、调用天气服务。

Agent 层负责把 AI 能力组织起来：什么时候读历史、什么时候查长期记忆、怎么拼 Prompt、怎么处理工具调用、怎么保存回复。

数据库层负责结构化数据，比如会话、消息、摘要、人设和心情记录。

向量库负责语义记忆，也就是把用户过去说过的重要内容转成向量，之后按语义相似度召回。

第三方工具负责外部信息，比如天气和热点新闻，这些不是 Agent 的核心推理能力，但能增强产品体验。

面试时可以强调：我这样拆不是为了“文件多”，而是为了让 HTTP、业务、AI 编排和数据持久化解耦。这样后续换模型、换向量库、加新工具、加用户体系，都不会把整个项目推倒重来。

## 4. 核心功能一：多轮对话是怎么实现的

多轮对话的核心不是简单把用户消息发给大模型，而是要处理三件事：会话隔离、历史上下文、消息持久化。

用户在前端选择一个会话后，前端会保存当前激活的会话 ID。每次发送消息时，请求体里会带上这个会话 ID 和用户输入内容。后端收到请求后，第一步会检查这个会话是否存在，如果不存在就返回错误，避免消息写到无效会话里。

确认会话存在后，后端会先保存用户消息。这里有一个细节：用户消息是在调用大模型之前保存的，这样即使模型调用失败，用户输入也不会丢。保存之后，Agent 会读取这个会话最近的几条消息，作为短期上下文传给模型。同时它还会读取这个会话最近一次压缩出来的摘要，作为中期记忆。

Prompt 的结构大概是这样：

```text
系统人设和回复规则
+ 工具调用说明
+ 可能的主动关怀提示
+ 本会话摘要
+ 最近几条原始消息
+ 当前用户消息
```

大模型生成回复后，后端再保存 assistant 消息。这样数据库里就形成了完整的 user / assistant 消息序列，前端刷新页面或者重新选择会话时，可以按时间顺序加载出来。

当前项目里没有真正的 `user_id`，也没有单独叫 `session_id` 的字段。它目前是单用户本地应用，使用会话 ID 来隔离不同聊天窗口。如果面试官追问多用户设计，可以这样回答：

> 当前版本里 conversation_id 承担了 session 的作用，保证不同会话的上下文隔离。真正上线时我会加 users 表，在 conversations 表里加 user_id 外键，并且所有消息查询、心情记录、RAG metadata 都带上 user_id，这样才能做到不同用户之间的数据隔离。

## 5. 核心功能二：为什么要设计三层记忆

情绪陪伴类项目最重要的一点是“它要像真的记得用户”。但大模型本身没有长期记忆，如果每次都只传当前问题，它就会忘记之前聊过的内容；如果把所有历史消息都塞进 Prompt，又会遇到上下文过长、成本上升和重点被稀释的问题。

所以这个项目用了三层记忆：

第一层是短期记忆。也就是最近几条原始聊天消息。它解决的是“当前对话能不能接得上”的问题，比如用户上一句说“他又这样了”，下一句问“我该怎么办”，模型需要知道“他”是谁。

第二层是中期记忆。当前项目设置了一个阈值，当某个会话消息数量超过一定数量后，会把最早的一批消息交给大模型压缩成一段摘要，然后保存到数据库里，再删除那批原始消息。这样既保留了关键信息，又不会让历史消息无限增长。

第三层是长期记忆。项目会把有价值的用户消息、心情总结和压缩摘要写入 ChromaDB。之后用户提到“上次那件事”“你还记得我之前说的吗”这类内容时，Agent 可以通过语义检索把相关历史找回来。

面试时可以这样讲：

> 我没有把历史消息粗暴地全塞进 Prompt，而是做了短期原文、中期摘要和长期向量记忆。短期负责对话连贯，中期负责压缩较长上下文，长期负责跨时间召回重要经历。这种设计能同时兼顾效果、成本和上下文窗口限制。

## 6. 核心功能三：RAG 在这个项目里不是文档问答，而是长期记忆

很多人一说 RAG 就会想到上传 PDF、切片、检索、回答文档问题。但这个项目里的 RAG 不是文档知识库型 RAG，而是情绪陪伴场景下的长期记忆 RAG。

项目里写入向量库的内容主要有三类：

1. 用户有实质内容的历史消息；
2. 被压缩后的会话摘要；
3. 每天生成的心情总结和关键词。

写入时，后端会调用 Embedding 模型把文本转成向量，再存到 ChromaDB。检索时，用户当前问题也会被转成向量，然后在 ChromaDB 里找语义最接近的历史内容。代码里还做了相似度过滤，低于一定阈值的结果不会注入 Prompt，避免无关记忆影响回复。

一个典型场景是：用户说“你还记得我之前说过那个同事吗？”模型第一次调用时会判断这句话需要查记忆，于是输出一个工具调用指令。后端解析出 query 后去向量库检索，把相关历史作为工具结果重新放进 Prompt，再让模型生成最终回复。

这比每次强行检索更自然。因为有些闲聊不需要查记忆，比如“今天好累”“晚上吃什么”，强行检索反而会打扰对话。当前实现让模型自己判断是否需要 `search_memory`，这就是一个轻量的 ReAct-like 设计。

需要注意的是，当前代码中没有文档上传型 RAG。也就是说，目前没有 PDF、Word、Markdown 上传，没有 chunk_size 和 overlap，也没有文件 MD5 去重和文件删除生命周期。如果面试官问，可以直接说：

> 当前项目的 RAG 重点是用户长期记忆，不是文档知识库。文档上传、切片、文件状态和删除同步目前还没有实现。如果要扩展，我会新增文件表、上传接口、解析器、分块策略，并在向量 metadata 中保存 file_id，删除文件时同步删除本地文件、数据库记录和向量索引。

## 7. 核心功能四：Agent 和工具调用是怎么做的

这个项目没有直接使用 LangChain Agent，也没有使用 LangGraph。它实现的是一个自定义的轻量 Agent 流程。

Agent 的核心思路是：先让模型判断是否需要工具，再由后端执行工具，最后把工具结果交回模型生成最终回答。

这里最重要的工具是 `search_memory`。系统 Prompt 里会告诉模型：如果用户明确提到之前聊过的事情，或者需要回忆历史背景，可以按固定格式输出工具调用。例如：

```text
[TOOL_CALL:search_memory:query=用户要检索的关键词]
```

后端会用规则解析这段内容。如果解析到了工具调用，就去 ChromaDB 检索相关历史；如果没有解析到，就认为模型不需要工具，直接把第一次回复流式返回给前端。

这个流程可以类比 ReAct：

```text
模型判断是否需要工具
    ↓
输出工具调用
    ↓
后端执行工具
    ↓
得到检索结果
    ↓
模型基于结果生成最终回答
```

为什么不用简单 if-else？因为用户表达历史记忆需求的方式非常多，不一定都会说“你还记得吗”。如果用关键词规则，很容易漏掉或者误判。让模型根据语义判断，灵活性更好。

如果以后要加新工具，比如日程提醒、附近地点查询、网页搜索，可以沿着同样的方式扩展：先写工具描述，让模型知道什么时候调用；再写后端执行函数；最后把工具结果注入模型上下文。

## 8. 核心功能五：流式响应是怎么实现的

聊天产品如果等大模型完整生成后再一次性返回，用户会明显感觉卡顿。所以这个项目使用了 SSE 流式响应。

后端的聊天流式接口会返回 `text/event-stream`。它不是普通 JSON，而是一段一段往前端推事件。事件大概有几类：

```text
[START]      表示流开始
[USER_MSG]   表示用户消息已经保存，并返回真实消息 ID
[TOOL_USING] 表示 Agent 正在检索记忆
普通 token    表示 AI 回复的一小段文本
[DONE]       表示回复结束，并返回 assistant 消息 ID
[ERROR]      表示过程中出错
```

前端不能用普通 axios 来处理这个流，所以使用 fetch 读取 ReadableStream。它会不断从后端读取 chunk，用 `TextDecoder` 解码，然后按 SSE 的 `data:` 格式解析。每收到一个 token，就追加到页面中那个临时 assistant 消息上。

这里有一个很重要的前端状态设计：用户点击发送后，前端会先乐观插入一条临时用户消息和一条空的临时 AI 消息，这样界面立刻有反馈。等后端返回 `[USER_MSG]` 和 `[DONE]` 后，再把临时消息 ID 替换成数据库里的真实 ID。

面试时可以这样说：

> 我自己设计了一套简单 SSE 事件协议，不只是传 token，还传用户消息保存成功、工具调用中、最终消息保存成功和错误状态。这样前端可以做到即时渲染、工具状态提示和数据库 ID 回填。

## 9. 核心功能六：心情总结和主动关怀怎么实现

心情总结的入口是用户点击“今日总结”。后端会读取指定日期的所有对话消息，但只拿用户说过的话去分析，不把 assistant 的回复算进去。然后它会让大模型按严格 JSON 格式输出三类信息：

```json
{
  "score": 7,
  "keywords": ["工作压力", "有些疲惫", "期待周末"],
  "summary": "今天你的状态整体..."
}
```

后端拿到结果后，会做几层处理：

1. 去掉模型可能返回的 Markdown 代码块；
2. 尝试解析 JSON；
3. 把 score 限制在 1 到 10；
4. 如果解析失败，就使用默认值兜底；
5. 保存到心情记录表；
6. 把这条心情总结写入 ChromaDB，作为长期记忆。

前端会把这些心情记录展示成几种形式：今日心情分数、近 7 天趋势、月历颜色块、关键词云，以及点击某天后的详情弹窗。

主动关怀逻辑是这个项目比较有产品感的一点。后端在构建聊天 Prompt 时，会检查最近三天的心情记录。如果最近三天都有记录，而且分数都低于 4，就会动态往系统 Prompt 里注入一段关怀提示，让 Agent 在回复时更自然地关注用户近况。

这说明项目不是只做“用户问一句，模型答一句”，而是会把结构化情绪数据反过来影响 Agent 的回复策略。

## 10. 数据库是怎么设计的

项目里有两类数据库。

第一类是关系型数据库，目前用的是 SQLite。它保存结构化数据，比如：

- 会话：标题、人设、创建时间、更新时间；
- 消息：属于哪个会话、角色是用户还是助手、消息内容、创建时间；
- 记忆摘要：属于哪个会话、摘要文本、压缩了多少条消息；
- 心情记录：日期、分数、总结、关键词、涉及哪些会话；
- 人设配置：人设名称、展示名、头像、系统 Prompt。

第二类是向量数据库，用的是 ChromaDB。它保存的不是业务表，而是语义向量。比如一条用户消息或者一条心情总结，会先通过 Embedding 模型变成向量，再存入 ChromaDB。后续检索时也是拿用户当前问题的向量去找相似内容。

为什么结构化数据不用向量库？因为会话列表、消息历史、心情日历这些数据需要精确查询、排序和关联，用关系型数据库更合适。

为什么长期记忆不用普通数据库 LIKE 查询？因为用户表达可能不完全一样。比如“之前那个同事让我很压抑”和“上次那个工作上的人”在关键词上不一定匹配，但语义上可能相关，这时向量检索更适合。

当前项目没有 MySQL。面试时可以说：

> 当前是本地原型，所以我用 SQLite 降低部署成本。表结构本身是用 SQLAlchemy ORM 定义的，未来迁移到 MySQL 或 PostgreSQL 成本不高，主要是把数据库 URL 配置化，并补迁移工具。

## 11. 前端是怎么和后端配合的

前端主要由三个区域组成：

左侧是会话侧栏，包括会话列表、新建会话、删除会话、人设切换、天气卡片和热点话题。

中间是聊天区域，包括历史消息加载、输入框、发送按钮、流式消息展示。

右侧是心情面板，包括今日心情、近 7 天趋势、月历、关键词云和心情详情弹窗。

普通接口用 axios 封装，比如获取会话、创建会话、查询心情记录、获取天气、获取新闻。聊天流式接口单独用 fetch，因为它需要逐 chunk 读取后端返回的 SSE 数据。

前端发送消息时，请求体很简单：

```json
{
  "conversation_id": 1,
  "content": "今天有点难受"
}
```

后端流式返回时，前端不是等完整 JSON，而是边收边处理。收到普通 token 就追加到 AI 消息里；收到工具调用状态就展示“正在翻阅记忆”；收到完成事件就更新真实消息 ID。

这样前后端之间不是简单请求响应，而是一个实时交互过程。

## 12. Prompt 设计应该怎么跟面试官讲

这个项目的 Prompt 不是只有一句“你是一个 AI 助手”，而是分层组织的。

第一层是人设 Prompt。项目里有温柔型、理性型、幽默型三种人设。每个会话会保存当前选择的人设，Agent 构建 Prompt 时会读取对应的人设配置。这样同一个后端逻辑可以支持不同风格的回复。

第二层是全局回复规则。它会约束模型不要像客服，不要长篇说教，不要过度 Markdown 排版，要像真实朋友一样简短自然地回应。这是情绪陪伴类产品很重要的体验控制。

第三层是工具调用说明。它告诉模型什么时候可以调用 `search_memory`，什么时候不应该调用，以及工具调用必须输出什么格式。

第四层是记忆上下文。包括本会话的压缩摘要、最近几条原始消息，以及必要时从 ChromaDB 检索到的长期记忆。

第五层是动态关怀 Prompt。它不是固定存在的，而是当最近几天心情分数持续较低时才注入。

面试官如果问“你怎么调 Prompt”，不要只说“我改了提示词”。可以这样回答：

> 我主要从四个方向调：第一是控制角色和语气，避免模型像客服或心理咨询模板；第二是控制输出长度，让回复更像聊天；第三是控制工具调用条件，避免每轮都查记忆；第四是控制结构化输出，比如心情总结必须返回 JSON，并在代码里做解析兜底。Prompt 调优不是只调一句话，而是结合产品体验和后端解析逻辑一起调。

## 13. 哪些技术没有实现，面试时怎么诚实回答

有些技术点在简历或面试中容易被追问，需要讲清楚边界。

当前项目没有实现 MySQL。当前用的是 SQLite，适合本地原型和演示。如果上线多用户生产环境，可以迁移到 MySQL 或 PostgreSQL。

当前项目没有使用 LangChain 和 LangGraph。Agent 是自定义实现的轻量流程，用大模型判断工具调用，再由后端执行工具。如果流程变复杂，比如多工具、多节点、多状态，可以用 LangGraph 重构。

当前项目没有实现 BM25。现在的长期记忆检索是向量检索。如果未来做专业知识库，BM25 可以补充精确关键词召回，再和向量检索做混合召回。

当前项目没有实现文档上传型知识库。没有 PDF、Word、Markdown 解析，没有 chunk_size 和 overlap，也没有文件 MD5 去重、文件状态表和文件删除同步。

当前项目没有完整用户体系。没有 user_id、登录注册和权限隔离，目前是单用户本地应用，通过 conversation_id 区分不同会话。

当前项目没有 Streamlit。前端是 Vue3 + TypeScript。

这样回答不会显得项目弱，反而显得你清楚项目边界。面试官通常更喜欢真实、准确、有优化思路的回答。

## 14. 高频追问与自然回答

### 1. 你这个项目和普通 ChatGPT 套壳有什么区别？

普通套壳通常只是把用户输入发给模型，再把结果返回。我的项目多了会话持久化、三层记忆、RAG 长期召回、心情总结、主动关怀和流式事件协议。也就是说，它不只是“问答”，而是围绕情绪陪伴做了产品闭环。

### 2. 为什么要保存用户消息后再调用模型？

这样即使模型调用失败，用户输入也不会丢失。对话系统里数据完整性很重要，用户说过的话应该先落库。后续也可以基于这些消息重新生成回复或做心情总结。

### 3. 为什么最近只取几条消息？

因为上下文窗口和成本都有限，所有历史都放进去会越来越重，而且会干扰当前问题。最近几条原文足够保证短期连贯，更早的内容通过摘要和 RAG 来补充。这样更可控。

### 4. 记忆压缩会不会丢信息？

会有一定信息损失，所以压缩 Prompt 会要求保留用户情绪状态、重要事件和个人背景。这个项目适合情绪陪伴，不需要逐字保留所有历史。真正重要的内容还会写入向量库，后续可以语义召回。

### 5. RAG 检索不准怎么办？

当前做了相似度阈值过滤，低相关内容不会进入 Prompt。工具说明里也告诉模型，如果检索结果不相关可以忽略。后续可以加 rerank、按用户过滤、混合召回和检索结果反馈。

### 6. 为什么不用每轮都检索记忆？

每轮都检索会增加延迟和成本，而且闲聊时容易把无关历史带进来。这个项目让模型先判断是否需要查记忆，只有在用户提到历史背景或类似经历时才调用工具。这样对话更自然。

### 7. 你的 Agent 是标准 ReAct 吗？

不是 LangChain 标准 ReAct，但思想类似。模型先判断是否需要工具，输出工具调用格式；后端执行工具；工具结果再进入模型上下文生成最终回答。我会把它称为轻量 ReAct-like 实现。

### 8. 为什么没有用 LangChain？

这个项目的工具链比较集中，核心就是一个记忆检索工具，所以自定义实现更轻量、可控。LangChain 适合更复杂的链式编排，但也会引入额外抽象。后续如果工具和状态变多，可以再引入 LangGraph。

### 9. 心情总结为什么只分析用户消息？

因为心情总结要反映用户状态，assistant 的回复不应该被算作用户情绪。如果把助手回复也放进去，模型可能分析到自己说过的话，导致情绪判断不准确。所以我只取用户消息。

### 10. LLM 返回的 JSON 不合法怎么办？

代码里会先去掉可能的 Markdown 代码块，然后尝试解析 JSON。如果解析失败，就使用默认 score、关键词和总结兜底。这样产品不会因为一次模型格式漂移直接崩掉。

### 11. 流式响应有什么难点？

难点在于它不是普通请求响应，前端要持续解析 chunk，还要维护临时消息状态。我的实现里会先插入临时用户消息和临时 AI 消息，收到后端事件后再更新真实 ID。还要处理工具调用状态和错误事件。

### 12. SSE 和 WebSocket 为什么选 SSE？

这个场景主要是服务端向客户端单向推送 AI 回复，SSE 就够用，实现也比 WebSocket 简单。用户发送消息还是普通 POST，请求之后后端持续返回 token。WebSocket 更适合双向实时通信，比如多人协作或实时游戏。

### 13. 为什么结构化数据用 SQLite，长期记忆用 ChromaDB？

会话、消息、心情记录需要精确查询和排序，用关系型数据库更合适。长期记忆需要按语义相似度查找，用向量库更合适。两者解决的是不同问题。

### 14. 向量库和数据库不一致怎么办？

当前实现是主数据优先，SQLite 写成功就算主流程成功，ChromaDB 写失败不会影响用户操作。这个选择适合原型，但生产环境要加补偿机制，比如索引状态字段、失败重试和定期一致性校验。

### 15. 怎么避免不同用户记忆串起来？

当前项目还没有用户体系，所以没有真正的 user_id 隔离。上线版本我会在所有会话、心情记录和向量 metadata 中加入 user_id，并在检索时按 user_id 过滤。这样才能避免跨用户记忆污染。

### 16. 为什么要做人设切换？

情绪陪伴不是只有一种表达方式。有些用户需要温柔接住，有些用户需要理性分析，有些用户喜欢轻松一点的表达。人设配置放在数据库里，切换会话人设后，Agent 构建 Prompt 时读取对应系统提示。

### 17. 主动关怀怎么避免让用户觉得被监控？

当前 Prompt 里会要求关怀要自然，不要让用户感觉被分析或监视。触发条件也比较克制，是连续三天低分才注入提示。更产品化的做法是给用户开关，让用户选择是否允许主动关怀。

### 18. 天气和新闻为什么放进这个项目？

它们不是核心 AI 能力，但能增加真实产品感。天气组件可以给用户生活建议，热点新闻可以作为聊天话题入口。这样产品不只是一个空白输入框，而是有更多自然互动入口。

### 19. 这个项目最大的技术难点是什么？

我认为是上下文和记忆管理。情绪陪伴需要连续性，但又不能把所有历史都塞进 Prompt。我的解决方案是短期原文、中期摘要、长期向量记忆，再由 Agent 判断是否检索。

### 20. 如果让你继续优化，你会先做什么？

我会先做用户体系和 RAG 隔离，因为这是从本地原型走向真实产品的基础。然后做统一异常处理、日志、测试和异步任务队列。AI 效果上会增加混合召回、rerank 和记忆管理界面。

## 15. 可以放进简历的项目亮点

- 独立实现情绪陪伴类 AI Agent 应用，完成从前端聊天交互、后端接口、消息持久化到 LLM 回复生成的完整链路。
- 设计短期上下文、中期摘要和长期向量记忆三层记忆机制，解决多轮对话中历史信息过长和长期记忆缺失的问题。
- 基于 ChromaDB 和 Embedding 构建长期语义记忆，将用户关键消息、心情总结和对话摘要沉淀为可检索记忆。
- 实现轻量 ReAct-like 工具调用机制，让 Agent 根据语义自主决定是否调用记忆检索工具，而不是固定规则检索。
- 使用 FastAPI 和 StreamingResponse 实现 SSE 流式聊天，前端通过 ReadableStream 逐 token 渲染 AI 回复。
- 基于 LLM 实现每日心情总结，生成情绪评分、关键词和总结文本，并在前端展示心情日历、近 7 天趋势和关键词云。
- 封装天气和新闻等第三方能力，把外部信息转化为聊天入口和生活建议，增强 AI 应用的产品完整度。

## 16. 最适合面试结尾的总结

这个项目我最想表达的是：我不只是会调用大模型 API，而是理解一个 LLM 应用从产品场景到工程实现需要哪些环节。比如聊天要有流式体验，长期陪伴要有记忆，记忆要考虑上下文窗口和召回准确率，心情总结要考虑结构化输出稳定性，数据库和向量库要考虑一致性，前后端还要处理状态同步。

如果继续推进，我会把它从“单用户本地 AI 应用”升级为“多用户可部署的 AI 陪伴系统”：补用户体系、权限隔离、异步任务、日志监控、Docker 部署、RAG 检索优化和记忆管理。这样它就不只是一个课程项目，而是一个更接近真实业务的 AI 应用原型。
