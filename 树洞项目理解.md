## 🚀 1. 项目全貌与核心概念 (Overview & Concepts)

### 这个项目到底是做什么的？

这是一个叫 **“树洞”** 的情绪陪伴应用。

用户可以在前端聊天界面里倾诉，后端会把消息交给 `CompanionAgent`，由它组织 Prompt、调用 LLM、必要时检索历史记忆，然后把回复以 **SSE 流式 token** 的方式返回给前端。系统还会把对话、记忆摘要、每日心情小记保存下来，让 Agent 不只是“每轮失忆地回答”，而是能逐渐形成对用户状态的上下文理解。

你可以把它想成这几层：

```text
Vue 前端
  -> FastAPI 路由
    -> CompanionAgent 对话编排
      -> LLM 服务
      -> SQLite 短中期记忆
      -> ChromaDB 长期语义记忆
```

### 核心概念 1：Agent 是“对话编排器”，不是单纯的一次 LLM 调用

核心类是 `backend/app/agents/companion_agent.py` 里的 `CompanionAgent`。

它负责：

- 保存用户消息：调用 `memory_service.save_message()`
- 检查是否需要压缩记忆：调用 `memory_service.check_and_compress()`
- 构建多层 Prompt：`CompanionAgent._build_prompt()`
- 第一次调用 LLM 判断是否需要工具：`CompanionAgent._invoke_llm_sync()`
- 如果需要，解析 `[TOOL_CALL:search_memory:query=...]`
- 调用 RAG 记忆检索：`CompanionAgent._execute_search_memory()`
- 第二次流式调用 LLM 并返回 token：`CompanionAgent._stream_llm()`
- 保存助手回复，并把有价值的用户消息写入长期记忆

所以新人读代码时要记住：**`CompanionAgent` 是主流程的大脑，`llm_service`、`memory_service`、`rag_service` 都是它调用的能力模块。**

### 核心概念 2：记忆被分成三种层次

项目里的“记忆”不是一个东西，而是三层：

1. **近期原始消息**
   - 存在 SQLite 的 `messages` 表，对应 ORM 类 `Message`
   - 由 `memory_service.get_recent_messages()` 取最近 5 条注入 Prompt

2. **中期摘要记忆**
   - 当单个会话消息超过阈值后，`memory_service.check_and_compress()` 会把早期消息压缩成一条摘要
   - 摘要存入 `memory_summaries` 表，对应 ORM 类 `MemorySummary`
   - 最新摘要由 `memory_service.get_latest_summary()` 注入 Prompt

3. **长期语义记忆**
   - 存在本地 ChromaDB：`backend/chroma_db/`
   - 由 `RAGService.index()` 写入，`RAGService.search()` 检索
   - 来源包括：用户有实质内容的消息、心情小记、压缩后的记忆摘要

一句话心智模型：**SQLite 管“事实存档”，ChromaDB 管“语义召回”，Prompt 管“本轮上下文”。**

### 核心概念 3：RAG 在这里不是每轮强制注入，而是 Agent 自主调用工具

很多 RAG 项目会在每次用户发消息时自动检索历史，然后把结果塞进 Prompt。这个项目不是这样。

这里的策略是：

- `MEMORY_TOOL_DESC` 会告诉 LLM：你有一个 `search_memory` 工具
- 第一次 LLM 调用只负责判断是否需要查记忆
- 如果 LLM 输出了类似：

```text
[TOOL_CALL:search_memory:query=用户之前提过的某件事]
```

后端才会进入 `CompanionAgent._execute_search_memory()`，调用 `RAGService.search()`。

这个设计的好处是：**普通闲聊不强行翻旧账，只有用户追问“你还记得吗”或明显需要历史背景时才检索。**

### 核心概念 4：异步感来自 SSE 流式返回

后端的 `CompanionAgent.stream_chat()` 是一个 Python Generator，它不断 `yield` 形如 `data: ...\n\n` 的 SSE 数据。

前端 `frontend/src/services/api.ts` 里的 `sendMessageStream()` 使用 `fetch()` + `ReadableStream` 消费这些数据，再由 `frontend/src/views/ChatView.vue` 把 token 逐字追加到消息气泡中。

重要事件包括：

```text
data: [START]
data: [USER_MSG]{...}
data: [TOOL_USING]
data: "一个 token"
data: [DONE]{...}
data: [ERROR]...
```

读懂这套协议，你就能读懂前后端聊天体验为什么是“边生成边显示”。

### 第 1 步：看数据模型，先知道系统存什么

打开 `backend/app/models/db_models.py`，重点看这 5 个 ORM 类：

这一阶段不需要理解业务细节，只要记住：**ORM 类管数据库，Schema 类管 API 数据形状。**

### 第 6 步：看记忆系统

## 🔍 4. 关键链路追踪 (Tracing the Core Flow)

这里追踪最重要的一条业务流：

**用户发起一次流式查询 -> 前端发送请求 -> FastAPI 路由接收 -> Agent 保存消息 -> 构建 Prompt -> LLM 判断是否调用记忆工具 -> RAG 检索 -> LLM 流式回复 -> 保存回复 -> 前端逐 token 渲染。**

### 总览流程

```text
ChatView.vue:send()
  -> api.ts:sendMessageStream()
    -> POST /api/chat/message/stream
      -> chat.py:send_message_stream()
        -> CompanionAgent.stream_chat()
          -> save_message()
          -> check_and_compress()
          -> _build_prompt()
          -> _invoke_llm_sync()
          -> _parse_tool_call()
          -> 可选：_execute_search_memory()
            -> _search_memory_with_fallback()
              -> RAGService.search()
              -> RAGService.format_for_prompt()
          -> _stream_llm()
          -> save_message()
          -> 可选：RAGService.index()
        -> StreamingResponse
    -> api.ts 解析 SSE
  -> ChatView.vue 更新 UI
```

## 🧰 5. 开发调试与贡献建议 (Debugging & Contribution Tips)

### 本地启动时先确认三件事

后端入口是：

```bash
cd backend
python run.py
```

前端入口是：

```bash
cd frontend
npm run dev
```

开发前请确认：

- `.env` 或环境变量里有可用的 `LLM_API_KEY` 或 `OPENAI_API_KEY`
- 如果要用天气和新闻，还需要 `AMAP_API_KEY`、`NEWS_API_KEY`
- 前端默认访问后端 `http://localhost:8000`，配置在 `frontend/src/services/api.ts`

### 新人第一次改代码，优先从这些小任务入手

1. **改前端展示文案**
   - 从 `frontend/src/views/ChatView.vue`、`MessageBubble.vue`、`ConversationSidebar.vue` 入手
   - 风险低，能快速熟悉组件结构

2. **加一个新的后端只读接口**
   - 参考 `backend/app/api/routes/memory.py`
   - 能熟悉 `Depends(get_db)`、Schema、SQLAlchemy 查询

3. **调整记忆检索策略**
   - 从 `CompanionAgent._search_memory_with_fallback()` 和 `RAGService.search()` 入手
   - 注意不要直接降低全局阈值，否则容易召回无关历史

4. **调整 Prompt 或人设**
   - 默认人设在 `backend/app/database.py` 的 `_insert_default_personas()` 中初始化
   - 运行过一次后，人设已存在 SQLite 中；只改代码不一定会更新现有数据库记录

### 常见坑位提醒

- **`CompanionAgent.chat()` 和 `CompanionAgent.stream_chat()` 是两套入口。**  
  前端主聊天使用的是流式接口，所以大多数体验问题应该优先看 `stream_chat()`。

- **第一次 LLM 回复不一定会直接展示给用户。**  
  在 `stream_chat()` 中，第一次同步调用主要用于判断是否需要工具；正式展示通常来自 `_stream_llm()`。

- **RAG 写入失败不一定会让主流程失败。**  
  `RAGService.index()` 内部捕获异常并返回 `False`，所以你要看控制台日志，不要只看接口是否 200。

- **ChromaDB 是本地持久化的。**  
  数据在 `backend/chroma_db/`，调试召回问题时要确认里面是否真的有索引。

- **SQLite 也是本地文件。**  
  数据库文件是 `backend/emotion.db`，模型定义在 `backend/app/models/db_models.py`。

- **当前 `chat.py` 的 `update_conversation()` 使用了 `ConversationUpdate`。**  
  阅读时注意它应该来自 `backend/app/models/schemas.py`。如果运行更新会话接口时报 `NameError`，优先检查 `backend/app/api/routes/chat.py` 顶部 import 是否包含 `ConversationUpdate`。

- **前端有临时消息 ID。**  
  `ChatView.vue` 会先用 `Date.now()` 生成临时用户消息和临时 assistant 消息，等后端返回 `[USER_MSG]` 和 `[DONE]` 后再替换成真实 ID。调试 UI 重复消息时要特别注意这里。

### 最后给新人的一句话

这个项目最值得先理解的不是“用了哪些库”，而是它对上下文的组织方式：

**短期上下文靠最近消息，中期记忆靠摘要，长期记忆靠 RAG，最终由 `CompanionAgent` 把它们按需编排成一轮自然的陪伴式回复。**

只要你抓住这条主线，剩下的文件都会慢慢归位。
